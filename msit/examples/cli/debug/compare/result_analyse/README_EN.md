# Procedure for analyzing the comparison result #

## Introduction ##

This section uses the dump data generated by the non-quantized Ascend AI processor as an example to describe how to compare the dump data generated by the non-quantized Ascend AI processor with the non-quantized Ascend model Npy data. The following uses this example as an example to describe the parameters. Replace them as required.

## Description of fields in the comparison result file ##

 *  **The comparison result is stored in the file.**`result_{timestamp}.csv`In, the meaning of the comparison result is the same as that of the basic precision comparison tool.  *  The following is a brief description of the results:

| OpType |      NPUDump | DataType | Address |   GroundTruth | DataType |           TensorIndex |            Shape | Overflow | CosineSimilarity | ... | MeanRelativeError | CompareFailReason | IsNpuOps | IsOutputNode | IsPrecisionError |

Pay attention to the following items in the preceding result file:

 *  \[x\] \[NPUDump\]: This corresponds to the operator in the OM model. Due to the fusion rule, multiple GPU/CPU operators may be corresponding.
 *  \[x\] \[DataType\]: There are two types of data. One is the data type on the NPU side, and the other is the data type on the CPU/GPU side. The two types are different and may cause precision loss.
 *  \[x\] \[GroundTruth\]: ONNX model operator corresponding to the om operator.
 *  \[x\] \[Overflow\]: indicates whether data overflow or underflow occurs.
 *  \[x\] \[CompareFailReason\]: indicates the cause of the comparison failure. The error may be calculated because the division by zero is invalid or does not match. As a result, the error becomes the NaN value. The detailed cause is listed.
 *  \[x\] \[IsNpuOps\]: specifies whether the node is unique to the NPU.
 *  \[x\] \[IsOutputNode\]: specifies whether the output node is the network-wide output node of the model.
 *  \[x\] \[IsPrecisionError\]: filtering nodes whose precision is abnormal.
 *  \[x\] \[CosineSimilarity\]\[RelativeEuclideanDistance\]...\[MeanRelativeError\]: This is the result of various error comparison types. You need to check whether a certain item exceeds the precision threshold (that is, a certain item is abnormal). If yes, pay special attention to it. ## Comparison Result Analysis ##

### Judging the comparison result ###

 *  Precision index

| Error comparison algorithm | Reference standard for normal precision |
| -------------------------- | --------------------------------------- |
| CosineSimilarity           | > 0.99                                  |
| RelativeEuclideanDistance  | < 0.05                                  |
| KullbackLeiblerDivergence  | < 0.005                                 |
| RootMeanSquareError        | < 1.0                                   |
| MeanRelativeError          | < 1.0                                   |

 *  **Whether the model precision meets the requirements depends on whether the output precision of the entire network meets the requirements. If the output precision meets the requirements, no action is required even if the precision of intermediate nodes is abnormal (including operator overflow). Otherwise, you need to check the faulty nodes one by one.**
 *  The precision of the front operator may affect the subsequent operators. You can enable the single operator comparison function (-single True) to check all the operators in the trouble ticket at a time and mask the impact of the front operator.

### Analysis of the comparison result ###

Analysis result provided by the tool:

 *  **The analysis result is printed on the screen after the invoking of the analyzer is complete. After the comparison is complete, the analyzer analyzes data line by line, excludes the nan data, and outputs the operator whose difference is not within the threshold range among the error comparison algorithms.**
    
    The output result is displayed in the markdown table.
    
    ```
    2023-04-19 13:54:10(1005)-[INFO]Operators may lead to inaccuracy:
    
    |                   Monitor |  Value | Index | OpType | NPUDump | GroundTruth |
    |--------------------------:|-------:|------:|-------:|--------:|------------:|
    |          CosineSimilarity | 0.6722 |   214 |    Mul |   Mul_6 |       Mul_6 |
    | RelativeEuclideanDistance |      1 |   214 |    Mul |   Mul_6 |       Mul_6 |
    ```
    
    For example, the first row in the preceding table indicates that the operator whose CosineSimilarity (cosine recognition degree) is less than 0.99 appears for the first time, which is the operator whose Index is 214, OpType is Mul, and the operator name is Mul_6 on the NPU side. Indicates the result of the cosine similarity comparison. The value range is \[-1, 1\]. If the comparison result is closer to 1, the values of the two are closer to each other. If the comparison result is closer to --1, the values of the two are more opposite.

## Expert advice on comparison results ##

Add the --advisor parameter as follows:

```
msit debug compare -gm /home/HwHiAiUser/onnx_prouce_data/resnet_offical.onnx -om /home/HwHiAiUser/onnx_prouce_data/model/resnet50.om \
-c /usr/local/Ascend/ascend-toolkit/latest -o /home/HwHiAiUser/result/test --advisor
```

 *  **FP16 overflow detection: The system performs overflow detection on the data whose data type is Float16 in the comparison data. If there is overflow data, expert advice is printed on the screen.**

 *  **Faulty node detection: Check whether a layer in the comparison result on the entire network is smaller than the threshold. If the subsequent data at the layer is smaller than the threshold, or the last layer is smaller than the threshold (cosine similarity < 0.99), the quantization error correction suggestions are provided.**
    
    For example, the following information is displayed:
